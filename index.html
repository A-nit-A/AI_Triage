AI TRIAGE

Construcción de un Agente de Triaje de Emails inteligente para TechPyme usando RAG y modelos locales.

El Problema de TechPyme

Una pequeña tienda online recibe decenas de correos al día. El dueño pierde 2 horas diarias respondiendo dudas básicas o derivando mensajes al departamento correcto manualmente.

Pérdida de Tiempo

Tareas repetitivas que no aportan valor estratégico al negocio.

Respuesta Lenta

El cliente espera demasiado tiempo por una duda sencilla de resolver.

Error Humano

Correos mal derivados o respuestas con información desactualizada.

Objetivo de la Actividad

Programar un agente inteligente que actúe como filtro inicial, respondiendo automáticamente usando las FAQs de la empresa o escalando a un humano si es necesario.

Arquitectura de la Solución

IA Local

Modelos de lenguaje (LLM) en local mediante Ollama. Privacidad 100% y cero costes por uso de API, bloqueando la fuga de datos de clientes corporativos.

Sistema RAG

Mecanismo de Generación Aumentada por Recuperación en ChromaDB. Inyecta inteligentemente la base de conocimiento (FAQs) de la empresa al LLM.

Agente

Centro lógico construido con LangChain. Recibe el email, coordina la búsqueda RAG, y toma la decisión de responder o escalar a un operador humano.

Uso de IA Local con Ollama

Para ejecutar el Agente de forma privada y sin costes, utilizaremos Ollama. Esta herramienta permite levantar modelos de lenguaje grandes (LLMs) directamente en nuestro equipo.

1. Instalar Ollama

Descarga el instalador según tu sistema operativo desde la página oficial (ollama.com) o instálalo por terminal en Linux/macOS:

# Instalar en Linux/macOS
curl -fsSL https://ollama.com/install.sh | sh

2. Descargar y Ejecutar

Abre una nueva terminal y ejecuta el siguiente comando para descargar e iniciar Llama 3 (un modelo muy capaz creado por Meta):

# Descarga y ejecuta Llama 3 (8B)
$ ollama run llama3

*La primera vez tardará unos minutos en descargar los pesos del modelo (~4.7GB).

Prompt de Sistema

El Prompt de Sistema es la directiva de comportamiento del agente. Define su identidad, restricciones y el protocolo de escalado. Su redacción es crítica.

System Prompt
"Eres un agente de soporte de TechPyme.
Responde preguntas de clientes usando solo el contexto
de las FAQs que se te proporciona.

Si la información solicitada NO está en el contexto,
o si el tono del cliente es agresivo o contiene insultos,

    [ESCALADO_HUMANO]

Pregunta del cliente: {question}"

# ─── Configuración del LLM y Retriever ───────────────────────────
from langchain_community.llms import Ollama
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate

# Prompt de sistema con contexto cerrado
SYSTEM_PROMPT = """Eres un agente de soporte de TechPyme.
Responde usando solo el contexto de las FAQs.
Si no tienes datos o hay agresividad, devuelve: [ESCALADO_HUMANO]
Contexto: {context}
Pregunta: {question}"""

prompt = PromptTemplate(
    template=SYSTEM_PROMPT,
    input_variables=["context", "question"]
)

qa_chain = RetrievalQA.from_chain_type(
    llm=Ollama(model="llama3", temperature=0.0),  # ← temperatura 0
    retriever=vectorstore.as_retriever(),
    chain_type_kwargs={"prompt": prompt}
)

Gestión de Alucinaciones

Las alucinaciones ocurren cuando el LLM inventa información no respaldada por el contexto. Dos mecanismos clave las mitigan:

Contexto Cerrado

Técnica: Grounding

La instrucción "usando solo el contexto" le prohíbe al LLM usar su conocimiento preentrenado. Si la FAQ no contiene la respuesta, el agente debe escalar, no inventar.

Temperatura = 0.0

Parámetro Crítico

Con temperature=0.0 el modelo siempre escoge el token más probable. Respuestas deterministas y predecibles. Nunca uses temperatura alta en agentes de producción.

0.0 · Preciso1.0 · Creativo2.0 · Caótico

Protocolo de Escalado Humano

Criterios de decisión para derivar el caso

Insultos o Agresividad

El agente detecta lenguaje hostil y transfiere automáticamente a un operador para preservar la relación con el cliente.

Información No Encontrada

La pregunta no tiene correspondencia en el RAG. El agente no inventa: escala antes que dar una respuesta incorrecta.

Casos Complejos o Legales

Reclamaciones, devoluciones complejas o temas legales superan la capacidad del agente y requieren criterio humano.

Cliente VIP o Recurrente

Clientes identificados como prioritarios pueden ser derivados siempre a un agente humano por política comercial.

El agente activa el escalado devolviendo la etiqueta exacta [ESCALADO_HUMANO] que el sistema Python detecta en tiempo real.

Concepto Avanzado

Arquitectura RAG 2.0

Más allá de la simple recuperación, el RAG Agéntico permite que el sistema razone sobre la calidad del contexto, corrija sus errores y decida dinámicamente qué herramienta usar.

RAG 1.0 (Lineal)

  • Confianza ciega en la primera búsqueda vectorial.
  • Propenso a alucinaciones si el contexto es pobre.
  • No entiende preguntas complejas o ambiguas.

RAG 2.0 (Agéntico)

  • Crítica y evaluación de la relevancia del contexto.
  • Reescritura inteligente de consultas (Query Translation).
  • Combinación de búsqueda semántica y de palabras clave.

Componentes del Agente RAG

1. Re-escritura (Query Translation)

El usuario pregunta: *"¿Mi paquete?"*. El agente traduce: *"¿Cuál es la política de tiempos de entrega y estado de envíos?"*. Esto optimiza radicalmente la precisión de ChromaDB.

2. Auto-Grader (Verificación)

Un LLM secundario actúa como "juez", analizando si los fragmentos recuperados realmente sirven para responder. Si no sirven, el agente no inventa; vuelve a buscar o escala.

3. Recuperación Híbrida

Combina lo mejor de dos mundos: Vectores (significado) + BM25 (exactitud). Crucial para encontrar códigos de error o nombres de productos específicos.
